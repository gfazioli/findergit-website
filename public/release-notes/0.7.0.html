FinderGit 0.7.0

Release notes

FinderGit 0.7.0 — the detail panel grows up

The right-hand detail panel splits into a real tabbed view, FinderGit
starts reading directly from GitHub, and the file browser gains three
new columns. Biggest release since 0.5.

A tabbed detail panel

Selecting a repo now opens a five-tab view in the right column:

  • Overview — GitHub-side identity: description, stars, forks,
    topic chips, latest release, top contributors
  • Unstage — the working-tree affordances that used to crowd the
    Overview: changes list, diff, commit form, AI button
  • Commits — recent local history with copy-SHA per row
  • Issues and Pull Requests — open items on the repo, with
    the same row shape and click-to-open behaviour

The global actions (Fetch / Pull / Push / Refresh) dock at the
bottom edge of the panel and stay reachable from every tab, so you
never leave Commits to push or leave Issues to pull.

The active tab is remembered across selections and across launches.

GitHub integration

FinderGit now reads Issues, Pull Requests and repository metadata
directly from your repos' GitHub remotes. Two paths to authenticate:

  • `gh` CLI — if you already ran `gh auth login`, FinderGit
    inherits the auth and there's nothing to set up.
  • Personal Access Token — fall back when `gh` isn't installed.
    Paste a token in Settings → Git → GitHub Token; FinderGit validates
    it before saving and stores it in your macOS Keychain (never on
    disk in FinderGit's own files).

If neither is configured, the GitHub-driven tabs and columns just
stay empty — the rest of the app keeps working.

Three new file-browser columns

Next to Changes, three GitHub-driven columns now appear: Issues
(open issues count), PRs (open pull requests count), and Stars
(stargazer count, with a small ⭐ icon). Click any column header to
sort — sort by Stars to surface your most-popular repo, by Issues to
find the busiest. Numbers use the compact `1.2K` format. Repos
without a GitHub remote leave the cells empty.

Commits tab is configurable

The new Commits tab defaults to 4 recent commits to keep the
panel skim-friendly. Crank it up to 50 in Settings → Git → Commit
History when you need more history.

Polish

  • Issue and PR rows now use GitHub-style icons (filled circle for
    open / closed issues; pull-request arrows for PRs with three
    states: open / merged / closed, plus a Draft pill) instead of
    plain coloured dots — a draft PR no longer looks identical to an
    open one.
  • Commit rows are tighter (gap dropped from ~25pt to ~13pt between
    rows) and the copy-SHA button is now vertically centred on each
    row with a green checkmark feedback for ~1 second after a copy.
  • The 'Git Only' toggle no longer flashes the unfiltered list for
    ~300ms when you switch root folders — the filter applies
    synchronously now.
  • Detail-panel height persists in the bottom layout, mirroring
    the width persistence that already worked in side-by-side mode.
    Drag the divider, switch repos, the height sticks.
  • Overview tab on local-only repos shows a short hint pointing at
    Unstage / Commits instead of an empty panel.

Under the hood

  • New `FinderGitCore` modules: `RemoteParser`, `RemoteProvider` (with
    CLI and PAT-backed implementations), `RemoteService` actor with
    per-listing TTL caches (60s for issues/PRs, 5min for file-browser
    counts), `KeychainStore` for host-keyed token storage.
  • File-browser counts driven by a single GraphQL query per repo so
    the three columns paint in one round-trip rather than three.
  • `RemoteHost` enum already recognises GitLab and Bitbucket
    hostnames; their providers ship in a follow-up release (#7).
  • 100% Swift Concurrency, no `DispatchQueue` in the new code; the
    cross-module enum switches carry `@unknown default` arms to stay
    Swift 6-ready.
